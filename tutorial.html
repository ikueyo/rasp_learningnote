1. 這是什麼專案

ImageGenCam 是一台自己組的數位相機：用樹莓派（Raspberry Pi）+ 小螢幕 + 相機模組組成，按下快門拍照後，把照片送給 AI 模型「重畫成某種風格」（例如把你畫成 Q 版貼紙），再存進相簿。手機可以連到相機看相簿、改 prompt。

它由幾個部分組成，這份手冊會逐一拆解：

  • 硬體：樹莓派、螢幕、相機、電池。
  • 韌體／程式：用 Python 寫的，跑在樹莓派上，開機自動啟動。
  • 生圖後端：可切換「雲端 Replicate」或「本地 ComfyUI」。
  • 手機 App：其實是相機自己開的一個網頁（PWA），手機連上同一個 Wi-Fi 就能用。

4. 一張照片的旅程（核心運作邏輯）

從「按下快門」到「相簿出現成品」，照片會經過這條流水線。看懂這張圖，就懂了這台相機的核心邏輯：

① 按 KEY1 快門
驅動層寫一個事件檔到 /tmp/imagegencam-shutter-events/
↓
② controller 收到 shutter 事件
把目前的相機畫面丟進 capture_queue（佇列）
↓
③ 拍照工人（capture worker）
把原圖存到 data/captures/，建立一個「生圖工作」存到 data/queue/generation/
↓
④ 生圖工人（generation worker）
呼叫 image_editor.edit_image(原圖, prompt, 輸出)
＝送雲端 Replicate 或本地 ComfyUI
↓
⑤ 成品寫進 data/generated/
相簿掃這個資料夾 → 顯示新照片
為什麼要分這麼多「工人」？因為生圖很慢（雲端要幾秒）。把它丟到背景工人去做，螢幕和快門才不會卡住，你可以繼續拍。這就是「多執行緒 + 佇列」的用途（第 6 章教）。
另外：Magic OFF 時，③ 會直接把原圖當成「成品」寫出去、跳過 ④，所以只拍照、不花錢。

5. 程式怎麼跑起來

開機後 systemd（Linux 的開機管理員）會執行 run.sh，它啟動 python3 -m imagegencam.app，也就是跑 app.py 的 main()：

def main() -> None:
    project_root = Path(__file__).resolve().parents[2]
    load_env_file(project_root / ".env")          # 1. 讀設定
    backend = os.environ.get("IMAGE_GEN_BACKEND", "replicate")
    if backend == "comfyui":                        # 2. 選後端
        image_editor = ComfyUIImageEditor(...)
    else:
        image_editor = OpenAIImageEditor(...)       # 其實是 Replicate
    controller = ImageGenCamController(... image_editor=image_editor ...)  # 3. 建主控
    web_server = WebServerThread(controller, host, port)
    web_server.start()                              # 4. 開手機網頁
    controller.run()                                # 5. 進主迴圈（不會回來，直到關機）

重點：controller.run() 是一個永遠在轉的迴圈。它每一圈都做這些事：

讀按鍵 → 把事件丟進 event_queue
↓
把 event_queue 裡的事件一個個交給 _handle_event() 處理（可能改變「目前畫面模式」）
↓
看現在是哪個「模式」(mode)，畫對應的畫面（預覽／選單／相簿／Wi-Fi…）
↺ 重複

這種「一直跑、每圈讀輸入再畫畫面」的設計叫 event loop（事件迴圈），遊戲和許多裝置都這樣寫。

5.1 看門狗：當主迴圈「凍住」怎麼自救

主迴圈萬一卡死（例如某次讀硬體 I2C 卡住、整圈不動了），會發生很麻煩的事：行程還活著、但什麼都不做。systemd 內建的「崩潰就重啟」只在行程死掉時才出手，這種「沒死但凍住」（軟當）它抓不到——曾經就這樣卡了 4 小時。

主迴圈每圈送一次「我還活著」
self._maybe_notify_watchdog(now) → 用 socket 送 WATCHDOG=1 給 systemd
↓ 如果迴圈凍住、停止報平安
systemd 在「外面」數秒
超過 WatchdogSec=30 沒收到 → 判定軟當
↓
自動殺掉 + 重啟
約 3 秒後相機 App 重起，從卡死中恢復
關鍵在「數秒的是 systemd，不是相機程式自己」。相機內建的 _check_stale_camera 跑在同一個迴圈裡，迴圈卡住它也跟著卡——救不了自己。看門狗是外部的旁觀者，所以抓得到。設定在 imagegencam.service 的 WatchdogSec=，報平安的程式在 controller.py 的 _sd_notify / _maybe_notify_watchdog。
觀念：內部 vs 外部監控。「自己監控自己」在自己當掉時也會失效；真正可靠的看門狗必須在被監控者之外。這是嵌入式/伺服器很常見的保命設計。

6. Python 教學（用本專案的真實程式）

下面用專案裡實際的程式碼，講你需要的 Python 觀念。

6.1 變數、函式、型別提示

def _format_web_url(self, host: str) -> str:   # host 是字串(str)，回傳也是字串
    if self.web_port == 80:
        return f"http://{host}"                  # f"..." 是字串內嵌變數
    return f"http://{host}:{self.web_port}"

host: str 與 -> str 是「型別提示」：告訴讀者（和工具）這裡放的是字串。Python 不強制，但能幫你少犯錯。f"http://{host}" 會把大括號裡的變數值塞進字串。

6.2 class（類別）與 self

class 是「把一堆資料和功能包成一個物件」。ImageGenCamController 就是一個大 class，代表「這台相機」。self 代表「這個物件自己」，用來存取它的資料：

class ImageGenCamController:
    def __init__(self, ...):          # __init__ 是建立物件時跑的「建構式」
        self.web_port = 8000          # self.xxx = 把資料記在這個物件身上
        self.wifi_manager = NetworkManagerWifi()
        self.running = True

    def shutdown(self):               # 物件的一個「方法」(method)
        self.running = False          # 之後 run() 的迴圈看到就會停
怎麼讀：看到 self.travel_hotspot_ssid 就是「這台相機記著的熱點名稱」；看到 self._enter_wifi_menu() 就是「叫這台相機去執行『進入 Wi-Fi 選單』這個動作」。開頭底線 _ 是慣例，表示「內部用、別從外面亂叫」。

6.3 dict 與 list（字典與清單）

self.button_lookup = {                              # dict：用「鍵」查「值」
    BUTTON_B: "toggle_generation",                  # 按鍵 → 事件名稱
    BUTTON_X: "ui_album",
}
action = self.button_lookup.get(pin)                # 用 pin 查出對應事件

POWER_ACTIONS = (                                    # tuple：固定不變的清單
    ("__power_shutdown__", "Shut Down", "poweroff"),
    ("__power_reboot__",  "Restart",   "reboot"),
)

prompt 清單就是存成這種結構（在 data/prompts.json），每筆有 id／title／body。

6.4 dataclass（資料類別）

當你只是想「裝一包欄位」，用 @dataclass 最省事。相機的「目前狀態」就是這樣：

@dataclass
class RuntimeState:
    mode: str = "preview"        # 目前畫面模式，預設 "preview"
    status_message: str = ""
    pending_jobs: int = 0
    ...

self.state.mode 決定畫面要畫什麼，是整個 UI 的核心開關（第 8 章）。

6.5 多執行緒 Thread、佇列 Queue、鎖 Lock（為什麼需要）

相機要同時做很多事：抓相機畫面、畫螢幕、等雲端生圖。如果排隊一件件做，生圖那幾秒會讓畫面凍住。解法是開背景執行緒各做各的：

# controller 建立時，開了好幾個背景工人：
self.capture_worker_thread = Thread(target=self._capture_worker_loop, daemon=True)
self.capture_worker_thread.start()
self.generation_worker_thread = Thread(target=self._generation_worker_loop, daemon=True)
self.generation_worker_thread.start()

工人之間用 Queue（佇列）傳工作，先進先出，安全：

self.capture_queue.put(request)    # 主迴圈：把工作放進佇列
job = self.capture_queue.get()     # 工人：從佇列拿工作（沒有就等）

多個執行緒可能同時改同一份資料（例如 self.state），會打架。用 Lock（鎖）保護，一次只准一個進去：

with self.state_lock:              # 進入時上鎖，離開時自動解鎖
    self.state.mode = "wifi_menu"
你新加的離家熱點功能就是一個背景工人：_travel_hotspot_monitor_loop 在背景每隔一段時間檢查「有沒有離線、掃不掃得到熱點」，要跳提示時就 with self.state_lock: self.state.mode = "wifi_hotspot_offer"。

6.6 try / except（例外處理）

硬體和網路常出錯。用 try/except 接住錯誤，避免整支程式崩潰：

try:
    result = image_editor.edit_image(source, prompt, output)
except Exception as exc:                      # 出錯時跑這裡
    logger.exception("Generation failed")     # 把完整錯誤寫進 log(journalctl 看得到)
    self.state.status_message = "Generation queued for retry."   # 螢幕顯示友善訊息,不崩潰

6.7 環境變數與 .env

不想把密碼、IP 寫死在程式裡，就放在 .env 檔，程式用 os.environ.get 讀：

backend = os.environ.get("IMAGE_GEN_BACKEND", "replicate")  # 沒設就用預設 "replicate"
self.travel_hotspot_ssid = os.environ.get("TRAVEL_HOTSPOT_SSID", "").strip()
踩過的雷：.env 會被 run.sh 用 bash source 載入。值若有空格或撇號，一定要用雙引號，例如 HOME_WIFI_SSID="Sunny 'n iku"；沒加引號會讓開機腳本解析失敗、服務崩潰。

6.8 import 與模組

from .wifi_manager import should_offer_travel_hotspot   # 從同資料夾的 wifi_manager 借函式來用
踩過的雷：controller.py 有這行 import，所以改了 controller 上 Pi 時，一定要連 wifi_manager.py 一起送，否則找不到那個函式 → 開機 ImportError → 崩潰迴圈。

7. 後端抽象：怎麼換生圖引擎

兩個後端 class（ReplicateImageEditor、ComfyUIImageEditor）都提供一模一樣的方法：

def edit_image(self, source_path, prompt, output_path, ...) -> Path:
    # 吃「原圖路徑 + prompt + 要輸出到哪」，回傳輸出檔路徑

controller 只認這個方法，不在乎背後是雲端還是本地。app.py 依 .env 決定建哪一個：

.env
IMAGE_GEN_BACKEND
app.py 選擇
replicate → 雲端
comfyui → 本地
controller
只呼叫
edit_image(...)
想加第三種後端？只要寫一個有 edit_image 的新 class，在 app.py 多一個分支即可，controller 一行都不用動。這就是「介面抽象」的威力。

8. 狀態機：按鍵如何變成畫面

相機任何時刻都處於某個「模式」(self.state.mode)，例如 preview prompt_picker album wifi_menu wifi_hotspot_offer。這叫狀態機：同一顆按鍵在不同模式做不同事。

preview 預覽
搖桿按下 → prompt_picker
→
prompt_picker 選單
選到 Wi-Fi → wifi_menu
選到 Shut Down → power_confirm
→
wifi_menu
選網路 → 輸密碼 → 連線

程式上有兩個對應點：

  1. 輸入：_handle_event(event) 裡用 if mode == "..." 決定這顆按鍵做什麼。
  2. 輸出：run() 主迴圈裡用 elif mode == "..." 決定要呼叫哪個 _render_xxx_frame() 畫畫面。
# 輸入：搖桿按下(ui_prompt) 在不同模式做不同事
if event == "ui_prompt":
    if mode == "prompt_picker":
        self._select_prompt_from_picker()
    elif mode == "wifi_hotspot_offer":
        self._confirm_travel_hotspot()      # 你新加的：確認連熱點
    ...

# 輸出：依模式畫對應畫面
elif mode == "wifi_hotspot_offer":
    self._render_travel_hotspot_offer_frame()
加一個新畫面的固定套路：①取一個 mode 名稱 → ②寫一個 _render_xxx_frame() 畫它 → ③在 run() 加一條 elif mode==… 呼叫它 → ④在 _handle_event 加按鍵處理 → ⑤寫一個 _enter_xxx() 設定 self.state.mode 進入它。你新增的「熱點詢問」畫面就是照這套做的。

9. 常見修改任務

新增／修改一個 prompt

改 data/prompts.json（每筆 id/title/body）。注意 config.py 裡有一份「內建種子」，測試會檢查兩者一致，所以兩邊要一起改（或直接用手機 App 改 prompt，不動檔案）。

改螢幕上的字／圖示

找對應的 _render_xxx_frame()（例如預覽畫面是 _build_preview_chrome_overlay），用 PIL 的 draw.text(...) / draw.rounded_rectangle(...) 畫。你「移除火花/相簿圖示」就是把那兩行 _draw_side_tab(...) 刪掉。

新增一個 .env 設定

在程式裡用 os.environ.get("我的設定", "預設值") 讀；在 .env.example 補一行說明；真實值寫進 Pi 上的 .env。

10. 維運指令與排錯（Local / Pi 自學版）

這章把日常維運分成兩邊——Local（家裡 PC / PowerShell）與 Pi（SSH 或 rpi-connect 進去後的 bash），每條指令都附「在學什麼」，再加「怎麼讀輸出」「錯誤解碼」，照著做也能邊用邊懂。

10.1 Local — 家裡 PC（PowerShell）

設 IP 變數（每個視窗先跑一次；之後用 ${PI} 代入）

部署檔案到 Pi（scp）　🔍 scp 來源 ikueyo@${PI}:目標路徑：冒號後是 Pi 上的目標資料夾，結尾 ` 是 PowerShell 換行。

# 在原始碼資料夾裡用裸檔名最簡單(避免 scp 把 C: 當主機名):

# 送多個檔(相依檔一起送,別漏新檔):
scp app.py controller.py config.py comfyui_client.py replicate_client.py prompt_framing.py `
    ikueyo@${PI}:~/imagegencam/software/src/imagegencam/

# 送資料檔(workflow):
cd C:\Users\Username\Desktop\imagegencam\compare-base\software\data
scp kontext.json ikueyo@${PI}:~/imagegencam/software/data/
⚠️ scp 的兩個雷：① 別直接 scp C:\...\file——scp 會把 C: 當成主機名；先 cd 到資料夾再用裸檔名。② 用相對路徑時，路徑從你目前所在資料夾算起；報 No such file 多半是人在子資料夾卻用了從根目錄起算的路徑。

連線 Pi / 找 IP　🔍 -t 配一個假終端機，讓遠端 sudo 有地方問密碼。

ping ikusrpb0614                 # 印出目前 IP(路由器有 DNS 時)
ssh ikueyo@${PI}                 # 一般連線
ssh -t ikueyo@${PI} "sudo systemctl restart imagegencam.service"   # 遠端跑 sudo 要加 -t
⚠️ 少了 -t 會報 sudo: a terminal is required。別用 ikusrpb0614.local（Windows 對 mDNS 支援差）。

ComfyUI 隧道（cloudflared，在 PC 上跑）　📘 cloudflared 跟 ComfyUI 同一台，所以連得到「只綁 127.0.0.1」的 ComfyUI，再開成公網網址給相機。

# 下載(只需一次,沒裝 winget 用這個):
Invoke-WebRequest -Uri "https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-windows-amd64.exe" -OutFile "$env:USERPROFILE\cloudflared.exe"

# 開 quick tunnel(視窗別關;印出 https://xxx.trycloudflare.com,每次都不同):
& "$env:USERPROFILE\cloudflared.exe" tunnel --url http://localhost:8188
⚠️ quick tunnel 網址每次都變、且無認證，只適合臨時用；開完要更新 Pi 的 COMFYUI_BASE_URL（見 10.3）。長期離家用要升 named tunnel + Cloudflare Access。

檢查 ComfyUI 監聽 / 本機開發驗證

netstat -ano | findstr :8188     # 看 ComfyUI 綁 127.0.0.1(只本機) 還是 0.0.0.0(對外)

10.2 Pi — SSH / rpi-connect 進去後（bash）

服務管理　🔍 systemctl 動作 服務名，--no-pager 一次印完。怎麼讀 status 見 10.4。

sudo systemctl restart imagegencam.service              # 重啟(改程式或 .env 後)
systemctl status imagegencam.service --no-pager         # 狀態
sudo systemctl stop imagegencam.service                 # 停(手動讀 I2C 前必停)
sudo systemctl start imagegencam.service                # 開

看 log　🔍 -u限定服務、-f即時、-n N最後N行、-p err只看錯誤、--since起始時間。

journalctl -u imagegencam.service -f                    # 即時跟看(拍照時盯,Ctrl+C 停)
journalctl -u imagegencam.service -n 120 --no-pager     # 最近 120 行
journalctl -u imagegencam.service -p err --no-pager     # 只看錯誤
journalctl -u NetworkManager --since "2026-06-21 12:00:00" --no-pager | tail -200
journalctl -k | grep -iE "i2c|spi|voltage"             # 核心的硬體/匯流排錯誤
⚠️ --since 不能寫 "today 12:00"（會 Failed to parse timestamp）——要用完整 "2026-06-21 12:00:00" 或相對 "6 hours ago"。

驗證 import（漏送檔/崩潰迴圈時）　📘 用 venv 的 python，系統 python3 沒裝依賴會誤報缺套件。

cd ~/imagegencam/software
PYTHONPATH=src .venv/bin/python3 -c "import imagegencam.app, imagegencam.prompt_framing; print('IMPORT_OK')"

改設定 .env（永遠在 Pi 上改，不 scp）

nano ~/imagegencam/software/.env
grep -iE 'comfyui|backend|wifi|ssid|hotspot' ~/imagegencam/software/.env   # 只看相關鍵
# 關掉沒用又會卡死的 PiSugar 0x57 輪詢(本機是 UPS-Lite,不需要它):
sed -i 's/^PISUGAR_POWER_BUTTON_SHUTTER=.*/PISUGAR_POWER_BUTTON_SHUTTER=0/' ~/imagegencam/software/.env
sudo systemctl restart imagegencam.service              # 改完一定要重啟

測 ComfyUI 連線　📘 回 JSON=通，把「相機連不到」拆成「網路問題還是程式問題」。

curl http://192.168.x.x:8188/system_stats              # 區網直連
curl https://xxx.trycloudflare.com/system_stats        # 走隧道

Wi-Fi（nmcli）　🔍 -t精簡、-f選欄、-g只印值。📘 nmcli 比相機螢幕鍵盤強：螢幕鍵盤打不出 ( ) + = ' " 等特殊字元，nmcli 用引號什麼密碼都能設。

nmcli -t -f active,ssid dev wifi                        # 目前連的是哪個(active=yes)
nmcli connection show                                   # 已存的連線(看 NAME / UUID)
nmcli device wifi list                                  # 掃到的網路(在不在範圍/2.4GHz)

sudo nmcli device wifi connect "WiFi名稱" password "密碼"      # 連線/存密碼(名稱密碼有空格用雙引號)
sudo nmcli connection modify "WiFi名稱" wifi-sec.psk "新密碼"  # 改密碼

# 確認密碼是「永久儲存」(重開機才不會掉,見 10.4):
sudo nmcli -g 802-11-wireless-security.psk-flags connection show "WiFi名稱"   # 要回 0
sudo nmcli connection modify "WiFi名稱" 802-11-wireless-security.psk-flags 0 wifi-sec.psk "密碼"

sudo nmcli connection modify "家裡WiFi" connection.autoconnect-priority 10    # 家裡優先於熱點

溫度 / 健康　📘 只有 SoC 有軟體溫度；UPS 板/電池要用紅外線測溫槍實體量。判讀見 10.4。

vcgencmd measure_temp                                   # SoC 溫度(<70°C 舒服)
vcgencmd get_throttled                                  # 0x0 = 沒被節流過
watch -n 2 vcgencmd measure_temp                        # 每 2 秒連續監看

電池 / I2C（手動查一定要先停服務）

sudo systemctl stop imagegencam.service
i2cdetect -y 1                  # 掃 I2C 裝置(這台的電量計在 0x32)
i2cget -y 1 0x32 0x04 w         # 讀 SOC(電量%)暫存器
sudo systemctl start imagegencam.service
服務跑著時手動 i2cget 會 Read failed——程式和你搶同一條 I2C 匯流排，正常現象，先 stop 服務再讀。

rpi-connect（遠端 shell；螢幕分享無桌面用不到）　📘 --user=使用者層級 systemd；enable-linger 讓它在你沒登入時也活著。

rpi-connect status                                      # 看登入/各功能狀態
loginctl enable-linger; rpi-connect on                  # 設一次:免登入也常駐
rpi-connect vnc off                                     # 關螢幕分享「政策」
systemctl --user disable --now rpi-connect-wayvnc.service   # 螢幕分享服務崩潰洗 log 時,徹底停掉
⚠️ vnc off 只關政策，底層 wayvnc 服務仍會崩潰重試；要 systemctl --user disable --now 才真停。螢幕分享需要桌面，無桌面的 Pi 用不到（也看不到相機畫面），管理用 remote shell 即可。

維護雜項

ls -l ~/imagegencam/software/src/imagegencam/controller.py   # 看檔案有沒有剛送上來(修改時間)
sudo chown -R ikueyo:ikueyo ~/imagegencam/software/data      # data 被 root 擁有導致 PermissionError 時

# 一次性:讓 log 重開機也不消失(持久化 journal)

10.3 常見情境流程（組合）

情境步驟
改了程式 → 部署① Local：py_compile+pytest → ② Local：scp 動到的檔(相依一起送) → ③ Pi：PYTHONPATH=src .venv/bin/python3 -c "import imagegencam.app" → ④ Pi：restart → status 確認
用 ComfyUI（quick tunnel）① PC：ComfyUI 開著 → ② PC：cloudflared tunnel --url http://localhost:8188，記新網址 → ③ Pi：nano .env 改 COMFYUI_BASE_URL → 重啟 → ④ Pi：curl 網址/system_stats 確認 → ⑤ 相機選單切 AI: ComfyUI。只想用雲端就跳過，切 AI: Replicate。
連不上 Pi① Local：ping ikusrpb0614 拿新 IP → ssh → ② 不行就用 connect.raspberrypi.com 的 Remote shell(免 IP) → ③ Pi 顯示離線=多半 Wi-Fi 斷,從相機螢幕 Wi-Fi 選單或熱點處理
改 workflow（步數/解析度/LoRA）① 改 data/kontext.json(或 ComfyUI 改後 Save API Format) → ② scp kontext.json 到 Pi 的 data/ → ③ 不用重啟,下次拍照就讀新的(仍須保持一個 LoadImage + 一個 CLIPTextEncode)

10.4 看懂輸出（怎麼讀結果）

systemctl status 三個關鍵欄位

Loaded: loaded (...; enabled; ...)   ← enabled=開機自啟 / disabled=不會自啟
Active: active (running) since ...    ← 正常;若 activating(auto-restart)=崩潰迴圈
Main PID: 1422 (bash)
⚠️ active (running) 只代表「外層 run.sh 活著」，不保證裡面 python 沒崩——還要看 log 有沒有 Traceback、restart 數字有沒有一直跳。

Wi-Fi 連線的「狀態機」（讀 NetworkManager log）。正常會依序經過：

prepare → config → need-auth →(有密碼)→ associating → associated → 4way_handshake → completed → ip-config → activated ✅
log 關鍵字意思
need-auth -> failed (reason 'no-secrets') / No agents available設定檔沒存密碼（psk-flags 不是 0）——無人在場重開機就連不上
4way_handshake failed / WRONG_KEY密碼錯（或路由器改了密碼）
卡在 associating / association took too long訊號弱 / AP 問題 / 被推去 5GHz（Pi 只有 2.4GHz）

vcgencmd 溫度與節流：measure_temp <70°C 舒服、80°C 軟節流、85°C 硬節流。get_throttled 0x0=從沒事；0x1正在低電壓、0x4正在降頻、0x8正在到溫度上限；高位（0x10000 起）=「曾經發生過」。

nmcli ... psk-flags 的值：0=系統永久儲存(重開機自動連)✅；1=agent-owned(要互動程式即時給→無人重開機連不上)⚠️；2=完全不存(每次問)。　netstat ...:8188：127.0.0.1:8188 LISTENING=只綁本機(要走隧道/改監聽)；0.0.0.0:8188=對外開放(區網可直連)。

11. 名詞表

名詞白話解釋
SSH從你的電腦用指令遠端登入樹莓派。ssh ikueyo@10.x.x.x
scp透過 SSH 把檔案複製到樹莓派。
systemd / serviceLinux 的「開機自動啟動 + 看門狗」。imagegencam.service 讓相機程式開機就跑、崩了自動重拉。
venvPython 的「獨立套件環境」。Pi 上程式跑在 .venv 裡，裝了 qrcode、Pillow 等；系統 python3 沒有。
NetworkManager / nmcliLinux 管 Wi-Fi 的工具。wifi_manager.py 就是包它的指令。
PWAProgressive Web App。相機開的網頁，手機可「加到主畫面」當 App 用，本質還是網頁。
mDNS / .local用主機名（如 imagegencam.local）找裝置。Windows 支援差，用 IP 最穩。
event loop / 事件迴圈一直轉的主迴圈，每圈讀輸入、更新畫面。
狀態機 (state machine)系統在「模式」之間切換；同一輸入在不同模式行為不同。
Thread / Queue / Lock背景執行緒／工作佇列／互斥鎖，用來「同時做很多事又不打架」。
Replicate / ComfyUI兩種生圖引擎：前者雲端付費，後者跑在你自己的 PC（免費但要開著）。
rpi-connect樹莓派官方遠端工具，免知道 IP 也能開遠端 shell（經 Cloudflare）。是 SSH 的免設定備援。
journal / journalctlsystemd 的統一日誌 / 看日誌的工具。服務的所有輸出都進這裡。
DHCP路由器自動分配 IP 的機制——所以 Pi 的 IP 會變，SSH 常要重查。
NAT多台裝置共用一個對外 IP。手機熱點的 NAT 較嚴格，P2P 連線（如 rpi-connect 螢幕分享）易卡。
tunnel / cloudflared把本機服務（只綁 127.0.0.1 的 ComfyUI）安全開成公網 https:// 網址的隧道程式。
SSID / PSKWi-Fi 的名稱 / 密碼。NetworkManager 用 psk-flags 決定密碼存不存（0=永久存）。
PYTHONPATH告訴 Python 去哪些資料夾找套件。本專案要設 src 才 import 得到 imagegencam。
SoC / throttle樹莓派主晶片（CPU+GPU 一體，measure_temp 測它）／ 過熱或電壓不足時自動降頻保護。
headless / Wayland·VNC無螢幕無桌面的機器（你的 Pi）／ 桌面顯示協定·遠端桌面；無桌面就用不到（螢幕分享失敗主因）。
